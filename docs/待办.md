# 待办

本文记录当前阶段已经完成后的下一步工作，按优先级排序。

背景约束：

- 当前 active / standby 同步功能还未上线，不需要考虑向下兼容。
- 当前已经完成 assignment generation 严格绑定。
- 当前部署前提仍然是每个实例本地挂载，各自有独立数据目录，不共享文件目录。

相关背景说明见：

- `docs/容灾方案.md`

## P0 立即推进

### 1. assignment 状态机落地

目标：

- 不再只是“active 续租一条 assignment”
- 而是让 assignment 真正反映复制生命周期

待做项：

- [ ] 明确并落地 assignment 状态迁移：`pending -> reconciling -> replicating -> draining -> released`
- [ ] 约束 generation 切换时机，避免新旧 active 或新旧 generation 并发写同一 standby
- [ ] `reconcile start` 与 assignment 状态联动，历史补齐进行中时明确标记为 `reconciling`
- [ ] 历史补齐完成后，将 assignment 切换为 `replicating`
- [ ] assignment 异常时写入 `error` / `last_error`，避免只在日志里丢失上下文

验收标准：

- 运维只看 assignment 状态，就能知道某个 standby 当前是在补历史、追增量、排水还是已释放

### 2. 让 assignment 成为唯一信号源

目标：

- 彻底去掉“配置、注册表、assignment 混合判断”的模糊性
- 所有复制行为都以有效 assignment 为准

待做项：

- [ ] 明确 active 侧 outbox / reconcile 只面向有效 assignment 目标工作
- [ ] 明确 standby 侧只接受当前有效 assignment 对应 active 的请求
- [ ] 进一步收紧 fallback 逻辑，避免在已有 control-plane assignment 时仍走静态配置思维
- [ ] 梳理 `peer_node_id` / `peer_base_url` 在 HA 模式下的角色，保留最小必要范围

验收标准：

- 复制目标、历史补齐目标、bootstrap 目标都能从 assignment 唯一推导，不依赖人工猜测

## P1 近期补齐

### 3. 旧 generation 数据清理与归档

目标：

- 既然 generation 已经严格生效，就要补上历史代际数据的生命周期管理

待做项：

- [ ] 为 `replication_outbox` 增加旧 generation 清理策略
- [ ] 为 `replication_reconcile_jobs` / `replication_reconcile_items` 增加保留期与清理策略
- [ ] 明确 `replication_offsets` 在 assignment 释放或切换后的保留策略
- [ ] 明确 `cluster_replication_assignments` 的 released / expired 记录是否归档或裁剪
- [ ] 设计人工触发与自动清理两种方式

验收标准：

- 长期运行后数据库不会因为历史 generation 累积而无限膨胀

### 4. reconcile 调度与自愈

目标：

- 不只支持“启动后一次”或“手工触发一次”
- 而是具备稳定的历史补齐自愈能力

待做项：

- [ ] 周期性 reconcile 调度器
- [ ] 启动后未完成 reconcile job 的恢复执行
- [ ] standby 长时间离线后恢复时，能够自动补历史缺口
- [ ] reconcile 失败重试与节流策略
- [ ] reconcile 差异比较与跳过逻辑，减少无效传输

验收标准：

- standby 中断、重启、长时间落后后，重新上线能够自动恢复到一致状态

### 5. 观测与告警

目标：

- 当前功能已经能跑，但还缺少“运维可持续观察”的能力

待做项：

- [ ] 复制 lag、failed events、reconcile pending items、assignment lease 状态指标
- [ ] generation mismatch、sequence conflict、assignment 失效等关键错误的结构化日志
- [ ] active / standby 心跳过期告警
- [ ] reconcile 失败与长期未完成告警
- [ ] `ha status` 输出进一步补充对 generation / assignment 状态的解释字段

验收标准：

- 不看数据库明细，也能通过日志、状态接口、指标快速定位复制链路问题

## P2 后续增强

### 6. 多 standby 编排

目标：

- 从“单 active 对单 standby”扩展到“单 active 对多个 standby”

待做项：

- [ ] 多 standby assignment 分配策略
- [ ] rebalance / drain / release 规则
- [ ] 多 standby 下的 reconcile 并发与限流
- [ ] 每个 standby 独立 generation / 状态观测

### 7. 运维 CLI 扩展

目标：

- 继续减少 curl、脚本和人工 SQL 操作

待做项：

- [ ] `warehouse ha assignments status` 增强筛选与详情输出
- [ ] `warehouse ha assignments drain`
- [ ] `warehouse ha assignments release`
- [ ] `warehouse ha assignments rebalance`
- [ ] reconcile / bootstrap / assignment 的统一观察入口

### 8. 大文件与性能优化

目标：

- 让历史补齐和增量复制在更大数据量下仍然可控

待做项：

- [ ] 大文件流式 reconcile
- [ ] 复制批量大小自适应
- [ ] 带宽限流与背压
- [ ] 单文件重复校验与跳过优化

## 当前建议执行顺序

1. 先做 assignment 状态机
2. 再做 assignment 唯一信号源收敛
3. 然后补 reconcile 调度与自愈
4. 再补观测、告警和清理策略
5. 最后推进多 standby 编排与运维 CLI 扩展

## 文档维护约定

以下场景发生变化时，需要同步更新本文：

- assignment 状态机变更
- generation 规则变更
- reconcile 调度方式变更
- 多 standby 能力落地
- 运维 CLI 能力新增
