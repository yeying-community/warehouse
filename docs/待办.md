## 现在就做

这部分是当前最值得优先投入的项，直接决定 standby 复制链路能否长期稳定跑。

### 1. reconcile 调度与自愈

目标：

- 在已具备“startup reconcile + periodic auto reconcile sweep”的基础上
- 继续补齐更完整的历史补齐自愈能力

待做项：

- [ ] 启动后未完成 reconcile job 的恢复执行
- [ ] reconcile 失败后的更细粒度节流、失败分类与分级重试策略
- [ ] reconcile 自动暂停阈值与恢复路径进一步细化
- [ ] reconcile 差异比较与跳过逻辑，减少无效传输

验收标准：

- standby 中断、重启、长时间落后后，重新上线能够自动恢复到一致状态
- 失败后的重试、退避、暂停、恢复路径可预测、可观察、可运维介入

### 2. 多 standby 下的 reconcile 并发与限流

目标：

- 在已经支持 `1 active -> N standby` 的基础上
- 补齐并发控制、带宽限制和资源保护

待做项：

- [ ] 多 standby 下的 reconcile 并发数控制
- [ ] 单 standby 与全局带宽限流
- [ ] active 本机资源保护策略，避免 reconcile 抢占正常写流量
- [ ] 单个 standby 异常时的资源隔离与退避策略

验收标准：

- 多个 standby 同时补历史时，不会明显拖垮 active 的正常服务
- 单个 standby 的慢或失败，不会放大成全局抖动

## 近期做

这部分是“主链路能稳定跑”之后，尽快补齐的运维能力和生命周期管理项。

### 3. 观测与告警

目标：

- 当前功能已经能跑，但还缺少“运维可持续观察”的能力

待做项：

- [ ] 复制 lag、failed events、reconcile pending items、assignment lease 状态指标
- [ ] standby 复制链路中 generation mismatch、sequence conflict、assignment 失效等关键错误的结构化日志继续补齐
- [ ] active / standby 心跳过期告警
- [ ] reconcile 失败与长期未完成告警
- [ ] `ha status` 输出进一步补充对 generation / assignment 状态的解释字段

验收标准：

- 不看数据库明细，也能通过日志、状态接口、指标快速定位复制链路问题

### 4. 旧 generation 数据清理与归档

目标：

- 既然 generation 已经严格生效，就要补上历史代际数据的生命周期管理

待做项：

- [ ] 为 `replication_outbox` 增加旧 generation 清理策略
- [ ] 为 `replication_reconcile_jobs` / `replication_reconcile_items` 增加保留期与清理策略
- [ ] 明确 `replication_offsets` 在 assignment 释放或切换后的保留策略
- [ ] 明确 `cluster_replication_assignments` 的 released / expired 记录是否归档或裁剪
- [ ] 设计人工触发与自动清理两种方式

验收标准：

- 长期运行后数据库不会因为历史 generation 累积而无限膨胀

## 后续做

这部分属于阶段三与后续增强，建立在前面稳定性、观测和生命周期管理已经补齐的基础上。

### 5. 多 standby 调度与编排增强

目标：

- 在已经支持 `1 active -> 多个 standby` 的基础上，继续补齐调度、切换和限流能力

待做项：

- [ ] rebalance / drain / release 的编排规则与保护策略
- [ ] failover / promote / 回切 SOP 与自动化边界

### 6. 运维 CLI 扩展

目标：

- 继续减少 curl、脚本和人工 SQL 操作

待做项：

- [ ] `warehouse ha assignments status` 增强筛选与详情输出
- [ ] `warehouse ha assignments rebalance`
- [ ] `warehouse ha assignments promote` / 切换辅助命令
- [ ] reconcile / bootstrap / assignment 的统一观察入口

### 7. 大文件与性能优化

目标：

- 让历史补齐和增量复制在更大数据量下仍然可控

待做项：

- [ ] 大文件流式 reconcile
- [ ] 复制批量大小自适应
- [ ] 带宽限流与背压
- [ ] 单文件重复校验与跳过优化

## 当前建议执行顺序

1. 先补 reconcile 调度与自愈
2. 再补多 standby 下的并发与限流
3. 随后补观测、告警和 generation 生命周期管理
4. 最后推进多 standby 调度、编排自动化与运维 CLI 扩展

## 文档维护约定

以下场景发生变化时，需要同步更新本文：

- assignment 状态机变更
- generation 规则变更
- reconcile 调度方式变更
- 多 standby 调度能力变化
- 运维 CLI 能力新增
- 待办优先级判断发生明显变化
- standby 相关核心术语或定义变化，例如 assignment、effective assignment、generation、fence、reconcile、bootstrap mark、watermark
