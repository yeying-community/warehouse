# S3 兼容接口 M0 验证记录

本文记录 [S3兼容接口实施清单.md](./S3兼容接口实施清单.md) 中 M0 技术验证的实际结果。

## 1. 验证环境

- 日期：2026-07-10
- Go：`go1.24.13 darwin/arm64`
- AWS Signature V4 signer：`github.com/aws/aws-sdk-go-v2 v1.42.1`
- AWS SDK for Go S3：`github.com/aws/aws-sdk-go-v2/service/s3 v1.105.0`
- AWS CLI：`aws-cli/1.45.45`
- rclone：`v1.74.4`
- region：`us-east-1`
- service：`s3`

本轮完成 AWS SDK for Go、AWS CLI、rclone 和 Go 反向代理验证。真实 Nginx 验证仍待执行。

## 2. 已实现验证器

代码位置：

```text
internal/interface/s3/signature_v4.go
```

当前支持：

- Header-based Signature V4。
- Credential Scope 解析。
- region 和 service 校验。
- 请求时间和最大时间偏差校验。
- SignedHeaders 排序、重复项和必需 Header 校验。
- canonical URI。
- canonical query。
- canonical headers。
- 普通 SHA-256 payload hash。
- `UNSIGNED-PAYLOAD` 显式开关。
- 明确拒绝尚未支持的 `STREAMING-*` payload。
- 常量时间签名比较。

当前验证器只验证签名中声明的 payload hash。

正式 PutObject 仍必须在流式写入临时文件时计算实际 SHA-256、MD5 和客户端声明的 `x-amz-checksum-*`，并在原子提交前校验。`UNSIGNED-PAYLOAD` 只跳过 SHA-256 声明值比较，不能跳过 Content-MD5 或 checksum 校验。

## 3. 固定签名向量

固定向量由 AWS SDK v2 signer 独立生成，Warehouse 测试中只保存最终请求和签名，不依赖 AWS SDK。

已覆盖：

- GET。
- PUT。
- 中文路径。
- 空格。
- `+`。
- `%2F`。
- 重复 query 参数。
- Content-Length。
- 请求 query 被篡改。
- region 不匹配。
- 请求时间过期。
- SignedHeader 缺失。
- streaming payload 暂不支持。
- rclone `UNSIGNED-PAYLOAD`、`Content-MD5` 和 `x-amz-meta-mtime`。

测试位置：

```text
internal/interface/s3/signature_v4_test.go
```

## 4. AWS SDK for Go PutObject 行为

使用 AWS SDK for Go 对 path-style Endpoint 发起小文件 PutObject。

观测结果：

```text
Method: PUT
Path: /personal/folder/%E6%B5%8B%E8%AF%95%20a%2Bb.txt
Query: x-id=PutObject
Content-Length: 8
Transfer-Encoding: none
Content-Encoding: none
X-Amz-Content-Sha256: 普通 SHA-256
```

SDK 还会把以下 Header 纳入 SignedHeaders：

- `accept-encoding`
- `amz-sdk-invocation-id`
- `amz-sdk-request`
- `content-length`
- `content-type`
- `host`
- `x-amz-checksum-crc32`
- `x-amz-content-sha256`
- `x-amz-date`

结论：

- 验证器不能只允许固定 Header 集合。
- 验证器必须按 SignedHeaders 动态读取并规范化 Header。
- AWS SDK for Go 的已知长度小文件 PutObject 不要求 aws-chunked。
- 大文件、未知长度 Reader 和 Multipart 行为仍需后续验证。

## 5. 反向代理验证

使用 Go `httputil.ReverseProxy` 模拟反向代理：

- 保留原始 Host。
- 保留 URI。
- 保留 raw query。
- 增加 X-Forwarded Header。

AWS SDK for Go 发出的真实 PutObject 经过代理后，Warehouse Signature V4 验证通过。

回归测试位置：

```text
internal/interface/s3/signature_v4_proxy_test.go
```

结论：

- 代理可以增加未被签名的 X-Forwarded Header。
- 代理不能修改 Host。
- 代理不能重写或重新编码路径。
- 代理不能改变 query 参数。
- 真实 Nginx 仍需使用目标配置单独验证。

## 6. AWS CLI 验证

使用 `aws-cli/1.45.45` 完成：

- ListBuckets。
- ListObjectsV2。
- PutObject。
- GetObject。

小文件 PutObject 使用普通 SHA-256 payload hash，并发送 `x-amz-checksum-crc32`。第一阶段 PutObject 必须在写入临时文件时同步计算并校验客户端声明的 checksum。

## 7. rclone 验证

使用 rclone `v1.74.4` 完成列桶、列对象、上传和下载。

观测到的默认行为：

- 使用 ListObjects v1，而不是 ListObjectsV2。
- 上传前对目标 bucket 调用 CreateBucket。
- PutObject 使用 `UNSIGNED-PAYLOAD`。
- 同时发送 `Content-MD5` 和 `x-amz-meta-mtime`。
- 对象响应缺少 `Last-Modified` 时提示 `Failed to read last modified`。

因此阶段 1 需要：

- 同时支持 ListObjects v1/v2。
- 对固定逻辑 bucket `personal`、`apps` 返回幂等 CreateBucket 成功，但禁止创建任意 bucket。
- 仅在直连 TLS 或可信代理确认的外部 HTTPS 请求中允许 `UNSIGNED-PAYLOAD`。
- 流式校验 Content-MD5。
- 特殊支持 `x-amz-meta-mtime` 并映射文件 mtime。
- HeadObject、GetObject 和列举响应返回 `Last-Modified`。

## 8. Signature V4 依赖评估

评估结果：

- AWS SDK for Go 提供 signer，但没有公开的服务端 verifier。
- `gofakes3 v1.2.0` 是完整模拟服务，不适合作为 Warehouse 的轻量协议适配依赖。
- VersityGW `v1.6.0` 是完整网关，并要求 Go 1.25。
- MinIO 的验签实现与完整服务内部结构耦合。

结论：保留 Warehouse 小型 Signature V4 verifier，并使用 AWS SDK 和实际客户端生成的固定签名向量持续回归。验证器只负责协议验签，对象权限、body checksum、原子写入和配额仍由 Warehouse 服务层处理。
