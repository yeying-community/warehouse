# 额度管理方案

本文定义 `warehouse` 当前用户存储额度（quota）管理的收敛方案，目标是让额度校验、容量统计、回收站口径、共享语义和 active / standby 边界在同一份文档里说清楚。

本文不是套餐系统设计，也不是多租户计费设计。第一阶段只解决“单用户存储额度可控、规则一致、实现可演进”。

## 1. 目标与非目标

### 1.1 目标

- 对用户上传、覆盖、复制等会增加逻辑存储占用的行为做额度校验
- 明确“已用空间”的统计口径
- 明确回收站、共享、active / standby 下的容量语义
- 让当前实现从“粗粒度可用”收敛到“规则更准确、后续可扩展”

### 1.2 非目标

- 不在第一阶段引入组织 / 租户 / 应用级多层额度
- 不在第一阶段做内容去重计费
- 不在第一阶段做完整套餐、账单、计费系统
- 不在第一阶段做强事务 reservation / 预占用状态机

## 2. 当前实现现状

当前仓库里已经存在一版基础额度能力：

- `users` 表已有：
  - `quota`
  - `used_space`
- WebDAV 写入口已有配额校验：
  - 位置：`internal/application/service/webdav_service.go`
- `GET /api/v1/public/webdav/quota` 已可返回当前用户额度信息
- 管理员接口已支持设置用户 `quota`

但当前实现仍有几个明显限制：

1. 上传前校验主要按“本次请求体大小”判断，没有精确区分“新建文件”和“覆盖已有文件”的增量差异
2. 写后 `used_space` 主要通过扫描用户目录重算，准确但成本较高
3. 回收站是否继续占用额度，虽然当前行为上等同“继续占用”，但规则没有在文档层明确定义
4. `COPY`、共享、active / standby 等场景的口径还没有统一成正式设计

因此，第一阶段不是“从零做 quota”，而是把已有实现收敛成一套更清晰、更稳定的方案。

## 3. 核心规则

### 3.1 额度主体

第一阶段只按“用户”做额度管理。

- `personal` 空间：按归属用户计费
- `apps` 空间：第一阶段仍按归属用户计费

后续如果要支持应用独立额度，再把主体抽象成：

- `owner_type`
- `owner_id`

第一阶段不提前引入这层复杂度。

### 3.2 已用空间统计口径

第一阶段的 `used_space` 表示：

- 当前用户逻辑文件内容占用的总字节数

不计入：

- 目录本身
- 数据库元数据表占用
- standby 副本磁盘占用
- 重复授权带来的“可见性变化”

### 3.3 回收站口径

第一阶段规定：

- 文件移入回收站后，**仍然占用额度**
- 只有“永久删除”时，才真正释放额度

原因：

- 回收站文件仍然真实占用磁盘
- 如果移入回收站就释放额度，会把回收站变成“免费冷存储”
- 与当前实现更一致，规则更简单

### 3.4 共享口径

共享、分组共享、公开分享都**不重复计费**。

规则统一为：

- 文件归谁，就算谁一次
- 分享只是授权访问，不是新副本
- 只有真正复制出一份新的逻辑文件时，才新增额度占用

### 3.5 unlimited 语义

当前实现里：

- `quota = 0` 表示不限额

第一阶段继续沿用这个语义，避免额外改动数据库和接口。

后续如果要提升可读性，可以再演进成显式字段，例如：

- `is_quota_unlimited`

## 4. active / standby 边界

这部分必须明确，否则很容易把 quota 做进复制链路，导致副本拒绝 active 的合法同步。

规则如下：

1. quota 只在 active 的对外写入口判定
2. standby 不独立做 quota 拒绝
3. standby 只接受来自 active 的合法复制请求
4. reconcile / outbox apply 不能重复增减 `used_space`
5. quota 真相以数据库中的用户额度字段为准，不以 standby 本地磁盘为准

一句话总结：

- quota 是“用户写入控制”
- 不是“副本复制控制”

## 5. 第一阶段落地策略

第一阶段不重构成全新的容量元数据系统，而是在当前 `users.quota + users.used_space` 的基础上做收敛。

### 5.1 写前校验

对会增加逻辑占用的请求，在写入前做额度判断。

第一阶段优先修正的核心规则是：

- 新建文件：按新文件完整大小判断
- 覆盖文件：只按 `delta = max(new_size - old_size, 0)` 判断
- `COPY`：按 `delta = max(source_size - target_existing_size, 0)` 判断

这意味着：

- 如果用户已有一个 `10MB` 文件
- 现在用 `12MB` 文件覆盖它
- 实际只应该新增 `2MB` 的额度校验压力

而不是再按 `12MB` 整体重复校验

### 5.2 写后刷新

当前实现仍然允许在写成功后刷新 `used_space`。

第一阶段继续保留这个策略，但要承认它是“简单可靠优先”的实现，而不是最终形态：

- 优点：实现简单、与现有代码兼容
- 缺点：每次写后扫描用户目录，成本较高

因此第一阶段的定位是：

- **先把写前判断做准**
- **写后继续重算兜底**

### 5.3 删除与回收站

第一阶段规则：

- `DELETE` 移入回收站：`used_space` 不减
- 回收站恢复：`used_space` 不变
- 回收站永久删除 / 清空回收站：`used_space` 减少

## 6. 后续演进方向

在第一阶段稳定后，再逐步演进以下能力。

### 6.1 增量更新替代全量扫描

后续目标：

- 对新建、覆盖、永久删除、回收站清理按 delta 直接更新 `used_space`
- 把当前“写后扫描全目录”降级为兜底修正手段

### 6.2 对账与修正

只做增量更新长期一定有漂移风险，因此后续要补：

- 周期性 quota 对账任务
- 手工 `check / rebuild` 命令
- 对账异常告警

### 6.3 更细粒度的对象元数据

如果后续要支持：

- 应用独立额度
- 更精确的回收站统计
- 更快的 quota rebuild

则需要补充更完整的逻辑文件元数据表，而不是只依赖目录扫描。

### 6.4 高并发下的 reservation

如果未来用户并发上传足够高，可能需要引入：

- 额度预占用
- 上传成功后提交
- 上传失败后回滚

第一阶段不做这层状态机。

## 7. 第一阶段最小开发清单

本轮建议先落地下面这组最小改动：

1. 补本文档
2. 修正 WebDAV quota 写前校验：
   - 覆盖已有文件时按大小增量判断
   - `COPY` 时按源路径与目标已存在路径的大小增量判断
3. 补测试覆盖：
   - 新建文件超额应拒绝
   - 覆盖已有文件时按 delta 判断
   - `COPY` 新建与覆盖目标时按 delta 判断
4. 保持当前 `quota = 0` 表示不限额
5. 保持当前“写后重算 `used_space`”策略不变

## 8. 暂不在本轮实现的内容

下面这些内容先沉淀在设计中，不在这一轮代码里一起做：

- 目录级 `COPY` / 目录树复制的更高性能 quota 估算
- quota 增量更新替代目录扫描
- 手工 `quota rebuild/check` CLI
- 更完整的文件容量元数据表
- active / standby 下的 quota 观测增强
- 多主体额度（用户 / 应用 / 组织）
