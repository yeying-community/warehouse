# 额度管理方案

本文定义 `warehouse` 当前用户存储额度（quota）管理的收敛方案，目标是让额度校验、容量统计、回收站口径、共享语义和 active / standby 边界在同一份文档里说清楚。

本文不是套餐系统设计，也不是多租户计费设计。第一阶段只解决“单用户存储额度可控、规则一致、实现可演进”。

## 1. 目标与非目标

### 1.1 目标

- 对用户上传、覆盖、复制等会增加逻辑存储占用的行为做额度校验
- 明确“已用空间”的统计口径
- 明确回收站、共享、active / standby 下的容量语义
- 让当前实现从“粗粒度可用”收敛到“规则更准确、后续可扩展”

### 1.2 非目标

- 不在第一阶段引入组织 / 租户 / 应用级多层额度
- 不在第一阶段做内容去重计费
- 不在第一阶段做完整套餐、账单、计费系统
- 不在第一阶段做强事务 reservation / 预占用状态机

## 2. 当前实现现状

当前仓库里已经存在一版基础额度能力：

- `users` 表已有：
  - `quota`
  - `used_space`
- WebDAV 写入口已有配额校验：
  - 位置：`internal/application/service/webdav_service.go`
- `GET /api/v1/public/webdav/quota` 已可返回当前用户额度信息
- 管理员接口已支持设置用户 `quota`

在最近一轮收敛后，当前实现已经具备：

1. `PUT/POST` 覆盖已有文件时，按大小增量做 quota 校验
2. `COPY` 新建 / 覆盖目标时，按大小增量做 quota 校验
3. 主写链路成功后，`used_space` 按 delta 更新，不再在正常写路径里每次扫描整个用户目录
4. 回收站永久删除 / 清空回收站时，会按记录中的 `size` 释放额度
5. CLI 已支持：
   - `warehouse quota check -c config.yaml --username <user>`
   - `warehouse quota rebuild -c config.yaml --username <user>`
6. 可选的后台自动对账任务已支持：
   - `quota.auto_reconcile_enabled`
   - `quota.auto_reconcile_interval`
   - 启用后会周期性重算所有用户的 `used_space`，自动修正漂移
   - 该任务默认关闭，只在非 `standby` 节点运行

当前仍有几个明显限制：

1. `COPY` 对目录树的大小估算还是直接递归扫描，性能上还不是最终形态
2. 更细粒度的对象元数据表还没引入，后续仍主要依赖路径扫描和回收站表
3. 自动对账当前是全用户定时扫描，规模更大时还需要更细的分片、限速和告警
4. 共享、active / standby 等场景的规则已经沉淀，但相关观测和运维辅助还不完整

因此，第一阶段已经从“粗粒度可用”收敛到“规则基本一致、可手工校正”，后续重点转向自动对账与更高性能的数据模型。

## 3. 核心规则

### 3.1 额度主体

第一阶段只按“用户”做额度管理。

- `personal` 空间：按归属用户计费
- `apps` 空间：第一阶段仍按归属用户计费
- `services` 空间：第一阶段仍按归属用户计费，不单独引入服务主体额度

后续如果要支持应用或服务独立额度，再把主体抽象成：

- `owner_type`
- `owner_id`

第一阶段不提前引入这层复杂度。

### 3.2 已用空间统计口径

第一阶段的 `used_space` 表示：

- 当前用户逻辑文件内容占用的总字节数

不计入：

- 目录本身
- 数据库元数据表占用
- standby 副本磁盘占用
- 重复授权带来的“可见性变化”

### 3.3 回收站口径

第一阶段规定：

- 文件移入回收站后，**仍然占用额度**
- 只有“永久删除”时，才真正释放额度

原因：

- 回收站文件仍然真实占用磁盘
- 如果移入回收站就释放额度，会把回收站变成“免费冷存储”
- 与当前实现更一致，规则更简单

### 3.4 共享口径

共享、分组共享、公开分享都**不重复计费**。

规则统一为：

- 文件归谁，就算谁一次
- 分享只是授权访问，不是新副本
- 只有真正复制出一份新的逻辑文件时，才新增额度占用

### 3.5 unlimited 语义

当前实现里：

- `quota = 0` 表示不限额

第一阶段继续沿用这个语义，避免额外改动数据库和接口。

后续如果要提升可读性，可以再演进成显式字段，例如：

- `is_quota_unlimited`

### 3.6 额度不足时的统一处理规则

当用户写入会导致额度超限时，系统遵循下面这组统一规则：

1. 拒绝本次新增占用写入
2. 不做“先写入再回滚”的隐式补偿
3. 不自动给用户扩容
4. 不因为存在 standby 副本而放宽判断

这里的“拒绝写入”包括但不限于：

- 上传新文件
- 用更大的文件覆盖已有文件
- `COPY` 复制出新的逻辑副本

这里的“通常不需要拒绝”包括：

- 删除到回收站
- 回收站恢复
- 不增加逻辑占用的元数据操作

一句话总结：

- 系统负责阻止超额写入
- 但不替用户决定“删数据还是扩容”

## 4. active / standby 边界

这部分必须明确，否则很容易把 quota 做进复制链路，导致副本拒绝 active 的合法同步。

规则如下：

1. quota 只在 active 的对外写入口判定
2. standby 不独立做 quota 拒绝
3. standby 只接受来自 active 的合法复制请求
4. reconcile / outbox apply 不能重复增减 `used_space`
5. quota 真相以数据库中的用户额度字段为准，不以 standby 本地磁盘为准

一句话总结：

- quota 是“用户写入控制”
- 不是“副本复制控制”

## 5. 第一阶段落地策略

第一阶段不重构成全新的容量元数据系统，而是在当前 `users.quota + users.used_space` 的基础上做收敛。

### 5.1 写前校验

对会增加逻辑占用的请求，在写入前做额度判断。

第一阶段优先修正的核心规则是：

- 新建文件：按新文件完整大小判断
- 覆盖文件：只按 `delta = max(new_size - old_size, 0)` 判断
- `COPY`：按 `delta = max(source_size - target_existing_size, 0)` 判断

这意味着：

- 如果用户已有一个 `10MB` 文件
- 现在用 `12MB` 文件覆盖它
- 实际只应该新增 `2MB` 的额度校验压力

而不是再按 `12MB` 整体重复校验

### 5.2 写后更新

当前实现已经切到：

- 主写链路成功后按 delta 更新 `used_space`
- 回收站永久删除 / 清空回收站时按 `recycle_items.size` 释放额度

这意味着：

- 正常 `PUT/POST/COPY` 不再每次都扫描整个用户目录
- `DELETE -> 回收站` 仍然不释放额度
- 只有真正永久删除时，才减少 `used_space`

当前这套实现的定位是：

- **主路径走增量更新**
- **异常漂移优先由后台自动对账修正，必要时仍可手工 `quota check/rebuild`**

### 5.3 删除与回收站

第一阶段规则：

- `DELETE` 移入回收站：`used_space` 不减
- 回收站恢复：`used_space` 不变
- 回收站永久删除 / 清空回收站：`used_space` 减少

### 5.4 额度不足时的处置闭环

除了“怎么统计、怎么校验”，还必须把“额度不够了谁来处理”固定下来，否则上线后问题会反复漂移到前端、后端、运维之间。

第一阶段的处置闭环定义如下。

#### 用户侧

用户是第一责任人，负责清理自己占用的逻辑数据。

当系统提示额度不足时，用户可执行的动作应该收敛为：

- 删除不需要的文件
- 清理回收站，做真正永久删除
- 减少复制行为，避免产生新的逻辑副本
- 联系管理员申请扩容

系统不应把“删除到回收站”描述成已经释放额度，因为这会误导用户。

#### 管理员侧

管理员是额度治理的直接责任人，负责判断用户是“真的额度不够”还是“统计有漂移”。

管理员的处理动作应收敛为：

- 查看该用户当前 `quota` / `used_space`
- 必要时执行 `warehouse quota check`
- 如果确认统计漂移，再执行 `warehouse quota rebuild`
- 如果确认业务上确实需要更多容量，再调整该用户 `quota`

也就是说：

- “修统计”属于管理员操作
- “加额度”也属于管理员操作

第一阶段不引入自动扩容、套餐扣费、审批流。

#### 系统侧

系统负责三件事：

1. 在写前准确判断是否超额
2. 在响应里明确告诉用户是“额度不足”
3. 通过自动对账和手工命令修复统计漂移

系统不负责：

- 自动给用户扩容
- 替用户自动删除旧文件
- 因为某个用户超额而暂停整个 active 服务

#### 运维侧

运维主要负责平台可用性，不直接替业务决定哪个用户该不该扩容。

运维动作通常只在以下场景介入：

- `used_space` 明显异常，需要执行 `quota check / rebuild`
- 自动对账任务过慢、失败或负载过高，需要调整周期
- 数据目录、回收站目录、数据库异常导致统计结果不可信

如果只是单个用户容量不够，优先由管理员判断是否扩容，而不是直接转成平台故障。

## 6. 后续演进方向

在第一阶段稳定后，再逐步演进以下能力。

### 6.1 对账与修正

只做增量更新长期一定有漂移风险，因此当前已经补上两层修正机制：

- 周期性 quota 对账任务
- 手工 `check / rebuild` 命令
- 对账异常告警

当前已经具备：

- `quota.auto_reconcile_enabled`
- `quota.auto_reconcile_interval`
- `warehouse quota check -c config.yaml --username <user>`
- `warehouse quota rebuild -c config.yaml --username <user>`
- 自动对账巡检到 `>= 80%` 或已超额用户时，会输出结构化日志，便于后续接日志告警

当前自动对账的重算口径与手工命令一致，都是：

- 用户目录下的实际文件大小总和
- 加上该用户 `recycle_items` 里的 `size` 总和

后续仍可继续补：

- 对账异常告警
- 分片扫描与限速
- 更细粒度的巡检观测

### 6.2 额度不足后的产品化能力

当前阶段已经明确：

- 用户负责清理数据
- 管理员负责修统计和扩容
- 系统只负责拦截与提示

后续如果要继续产品化，可以按下面顺序演进：

1. 使用率预警
   - 例如 80% 提醒、90% 强提醒、100% 拒绝新增写入
2. 管理端的额度视图
   - 支持按使用率、已超额、接近上限筛选用户
3. 用户自助申请扩容
   - 但是否通过，仍由管理员决定
4. 套餐/计费系统
   - 这已经超出第一阶段范围

### 6.3 更细粒度的对象元数据

如果后续要支持：

- 应用独立额度
- 更精确的回收站统计
- 更快的 quota rebuild

则需要补充更完整的逻辑文件元数据表，而不是只依赖目录扫描。

### 6.4 高并发下的 reservation

如果未来用户并发上传足够高，可能需要引入：

- 额度预占用
- 上传成功后提交
- 上传失败后回滚

第一阶段不做这层状态机。

## 7. 第一阶段最小开发清单

本轮建议先落地下面这组最小改动：

1. 补本文档
2. 修正 WebDAV quota 写前校验：
   - 覆盖已有文件时按大小增量判断
   - `COPY` 时按源路径与目标已存在路径的大小增量判断
3. 补测试覆盖：
   - 新建文件超额应拒绝
   - 覆盖已有文件时按 delta 判断
   - `COPY` 新建与覆盖目标时按 delta 判断
4. 主写链路成功后按 delta 更新 `used_space`
5. 增加手工 `quota check / rebuild` CLI
6. 增加可选的自动额度对账任务
7. 保持当前 `quota = 0` 表示不限额
8. 明确额度不足时的用户 / 管理员 / 系统 / 运维职责边界

## 8. 暂不在本轮实现的内容

下面这些内容先沉淀在设计中，不在这一轮代码里一起做：

- 目录级 `COPY` / 目录树复制的更高性能 quota 估算
- 更完整的文件容量元数据表
- active / standby 下的 quota 观测增强
- 多主体额度（用户 / 应用 / 组织）
- 使用率预警、申请扩容、套餐/计费等更完整产品化能力
