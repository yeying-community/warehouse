## 1. 部署对象

- 服务名称：`warehouse`
- 部署环境：开发、测试、预发、生产
- 安装包名称规则：`warehouse-<tag>-<short-hash>.tar.gz`
- 启动入口：`scripts/starter.sh`

本部署文档只面向“安装包部署”场景，不覆盖开发目录直接启动。

## 2. 安装包信息

### 2.1 安装包生成方式

在仓库根目录执行：

```bash
bash scripts/package.sh
```

如果需要按已有 TAG 重新打包：

```bash
bash scripts/package.sh v0.0.19
```

### 2.2 安装包文件名示例

```text
warehouse-v0.0.19-40cc83f.tar.gz
```

### 2.3 安装包位置

```text
output/<package-name>.tar.gz
```

### 2.4 安装包内容

当前安装包由 `scripts/package.sh` 生成，解压后至少包含：

```text
<package-dir>/
├── bin/warehouse
├── config.yaml.template
├── scripts/starter.sh
├── web/
└── resources/
```

## 3. 目标环境准备

- 操作系统：建议 Linux 服务器，例如 Ubuntu 22.04+ / Debian 12+
- CPU / 内存 / 磁盘：
  - CPU：2 核起
  - 内存：4 GB 起
  - 磁盘：按 `webdav.directory` 数据量规划，建议业务数据与系统盘分离
- 运行时依赖：
  - PostgreSQL 可达
  - `bash`
  - `nohup`
  - `tar`
- 部署目录：例如 `/opt/deploy`
- 运行账号：建议专用非 root 账号；如使用 root，需自行控制目录权限

建议预先创建部署目录：

```bash
mkdir -p /opt/deploy
```

如果要启用阶段一高可用复制，还要提前准备：

- 每个实例唯一的 `node.id`
- 其他实例可访问的 `node.advertise_url`
- 一致的 `replication.shared_secret`
- 每台机器自己的本地数据目录挂载点

## 4. 部署步骤

### 4.1 获取安装包

本地确认已生成安装包：

```bash
ls output/
```

### 4.2 拷贝安装包到目标机器

```bash
scp output/warehouse-v0.0.19-40cc83f.tar.gz <user>@<host>:/opt/deploy/
```

### 4.3 登录目标机器并解压

```bash
ssh <user>@<host>
cd /opt/deploy
tar -xzf warehouse-v0.0.19-40cc83f.tar.gz
cd warehouse-v0.0.19-40cc83f
```

### 4.4 初始化配置

安装包内当前提供的是 `config.yaml.template`，部署时复制为 `config.yaml`：

```bash
cp config.yaml.template config.yaml
```

至少需要修改这些配置：

- `database.*`
- `webdav.directory`
- `web3.jwt_secret`

如果启用阶段一高可用复制，还必须修改：

- `node.id`
- `node.role`
- `node.advertise_url`
- `replication.enabled`
- `replication.shared_secret`

建议重点确认这些复制相关配置：

- `replication.retry_backoff_base`
- `replication.max_retry_backoff`
- `replication.reconcile_auto_pause_failures`

说明：

- `retry_backoff_base` / `max_retry_backoff` 同时影响 outbox 重试和 assignment error 自动恢复节奏
- `reconcile_auto_pause_failures` 默认 `3`
- 设为 `0` 表示关闭自动暂停

如需生成随机密钥，可使用：

```bash
openssl rand -base64 48
```

适合用于：

- `web3.jwt_secret`
- `replication.shared_secret`

### 4.5 启动服务

```bash
bash scripts/starter.sh
```

或：

```bash
bash scripts/starter.sh start
```

### 4.6 停止服务

```bash
bash scripts/starter.sh stop
```

### 4.7 重启服务

```bash
bash scripts/starter.sh restart
```

## 5. 验证方式

### 5.1 启动结果验证

查看进程与日志：

```bash
cat run/warehouse.pid
tail -f logs/warehouse.log
```

### 5.2 健康检查验证

```bash
curl http://127.0.0.1:6065/api/v1/public/health/heartbeat
curl http://127.0.0.1:6065/api/v1/public/health/readiness
```

如果是 HTTPS 或反向代理，请替换为实际域名。

### 5.3 端口监听验证

```bash
ss -lntp | grep 6065
```

### 5.4 高可用复制验证

如果启用了 replication，可进一步执行：

```bash
./bin/warehouse -c config.yaml ha status
./bin/warehouse -c config.yaml ha assignments status
```

重点关注：

- assignment 是否存在
- `state` 是否处于 `pending / reconciling / replicating`
- `failureCount`
- `nextRetryAt`
- 是否已经进入 `paused`

## 6. 回滚方式

建议部署时保留至少一个旧版本目录，例如：

```text
/opt/deploy/
├── warehouse-v0.0.18-1234567/
└── warehouse-v0.0.19-40cc83f/
```

### 6.1 停止当前版本

```bash
cd /opt/deploy/warehouse-v0.0.19-40cc83f
bash scripts/starter.sh stop
```

### 6.2 切回旧版本

```bash
cd /opt/deploy/warehouse-v0.0.18-1234567
bash scripts/starter.sh start
```

### 6.3 恢复旧配置

如果新旧版本配置项不完全一致，回滚前应恢复旧版本自己的 `config.yaml`，不要直接混用新版本配置。

### 6.4 回滚后验证

```bash
curl http://127.0.0.1:6065/api/v1/public/health/readiness
tail -n 100 logs/warehouse.log
```

## 7. 常见问题

### 7.1 配置文件缺失

现象：

- 启动时报 `config not found`

处理：

```bash
cp config.yaml.template config.yaml
```

然后补齐目标环境配置。

### 7.2 启动脚本执行失败

现象：

- `scripts/starter.sh` 启动后立即退出

处理：

- 检查 `bin/warehouse` 是否存在且可执行
- 检查 `config.yaml` 是否存在
- 查看 `logs/warehouse.log`

### 7.3 端口冲突

现象：

- 服务无法监听 `6065`

处理：

```bash
ss -lntp | grep 6065
```

修改 `config.yaml` 的 `server.port`，或释放冲突端口。

### 7.4 数据目录权限不足

现象：

- 启动时报 `webdav.directory` 不可写

处理：

- 检查部署账号对数据目录是否有读写权限
- 确认挂载目录已存在，且权限正确

### 7.5 assignment 自动进入 paused

现象：

- `ha assignments status` 里某条 assignment 进入 `paused`

处理：

- 先检查 `lastError`、`failureCount`、`nextRetryAt`
- 排查 `node.advertise_url`、`replication.shared_secret`、网络与 internal 鉴权
- 修复后执行：

```bash
./bin/warehouse -c config.yaml ha assignments resume --standby-node-id <standby-node-id>
```
