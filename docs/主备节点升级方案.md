# 主备节点升级方案

本文给出 `warehouse` 在 `1 active + 1 standby` 架构下的升级操作顺序，以及升级后数据同步状态检查命令。

## 1. 目标与原则

- 优先保证业务连续性：先升级 `standby`，验证正常后再升级 `active`。
- 升级窗口内允许短时 lag，不强求 standby 立刻追平 active。
- 升级完成后要求复制链路持续前进，不出现长期卡住。

## 2. 升级前检查

在 active 节点执行：

```bash
cd /opt/deploy/warehouse-<version>
bin/warehouse ha status -c config.yaml --target-node-id warehouse-standby
bin/warehouse ha assignments status -c config.yaml --all
```

关注点：

- `resolvedPeerHealthy=true`
- assignment 存在且 `leaseExpired=false`
- 记录当前 `resolvedGeneration`、`lastAppliedOutboxId`、`lastOutboxId` 作为基线

## 3. 推荐升级顺序

### 3.1 先升级 standby

1. 在 standby 节点停止旧版本服务。
2. 部署新版本目录（例如 `/opt/deploy/warehouse-vX.Y.Z`），注意更新新版本目录下的`config.yaml` 文件。
3. 启动 standby 服务。
4. 在 active 节点确认 standby 心跳恢复。

检查命令（active）：

```bash
cd /opt/deploy/warehouse-vX.Y.Z-*
bin/warehouse ha status -c config.yaml --target-node-id warehouse-standby
```

### 3.2 再升级 active

1. 在 active 节点停止旧版本服务。
2. 部署新版本目录，注意更新新版本目录下的`config.yaml` 文件。
3. 启动服务。
4. 启动后观察 assignment 和 replication 状态。

检查命令（active）：

```bash
cd /opt/deploy/warehouse-vX.Y.Z
bin/warehouse ha assignments status -c config.yaml --all
bin/warehouse ha status -c config.yaml --target-node-id warehouse-standby
bin/warehouse ha reconcile status -c config.yaml --target-node-id warehouse-standby
```

## 4. 升级后同步状态检查命令

## 4.1 基础状态

```bash
# active 本机状态
bin/warehouse ha status -c config.yaml

# active 观察指定 standby
bin/warehouse ha status -c config.yaml --target-node-id warehouse-standby

# assignment 控制面
bin/warehouse ha assignments status -c config.yaml --all

# reconcile 任务状态
bin/warehouse ha reconcile status -c config.yaml --target-node-id warehouse-standby
```

## 4.2 判定“同步未卡住”的最小标准

满足以下条件即可判定链路在持续同步：

- `lastAppliedOutboxId` 随时间持续增长
- `pendingEvents` 不持续单边上涨
- `lastError` 不长期固定在同一条 fence/顺序冲突错误

说明：standby 不必实时追平 active，但不能连续较长时间（例如 5~10 分钟）完全不前进。

## 5. 常见异常与处置

### 5.1 generation fence 未初始化

典型错误：

- `standby generation fence is not initialized`
- 报文中出现 `assignmentGeneration != offsetGeneration`

在 active 执行：

```bash
bin/warehouse ha bootstrap mark -c config.yaml --peer --target-node-id warehouse-standby
```

再检查：

```bash
bin/warehouse ha status -c config.yaml --target-node-id warehouse-standby
```

### 5.2 outbox 顺序冲突

典型错误：

- `expected next outbox id N, got a later event after M`

优先处理：

1. 先观察是否自动恢复（短时间重试后可能自愈）。
2. 若持续出现，执行一次 reconcile。

```bash
bin/warehouse ha reconcile start -c config.yaml --target-node-id warehouse-standby --timeout 300s
```

若本机回环地址不稳定（`127.0.0.1` 超时/EOF），使用显式地址：

```bash
bin/warehouse ha reconcile start -c config.yaml \
  --base-url http://192.168.0.101:6065 \
  --target-node-id warehouse-standby \
  --timeout 300s
```

### 5.3 assignment 状态长时间停在 reconciling

在 active 执行：

```bash
bin/warehouse ha assignments status -c config.yaml --all
```

若长期不变且复制不前进，可按运维窗口执行链路重建：

```bash
bin/warehouse ha assignments release -c config.yaml \
  --active-node-id warehouse-active \
  --standby-node-id warehouse-standby \
  --force

bin/warehouse ha bootstrap mark -c config.yaml --peer --target-node-id warehouse-standby
```

然后继续观察 `ha status` / `ha reconcile status`。

## 6. 升级完成验收

在 active 节点连续执行 2~3 次（间隔 1 分钟）：

```bash
bin/warehouse ha status -c config.yaml --target-node-id warehouse-standby
bin/warehouse ha assignments status -c config.yaml --all
```

验收通过条件：

- standby 心跳健康（`resolvedPeerHealthy=true`）
- assignment 存在且 lease 未过期
- `lastAppliedOutboxId` 持续前进
- 无持续重复的同类致命错误（尤其 generation fence 冲突）
