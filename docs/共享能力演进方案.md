# 共享能力演进方案

本文不是当前实现说明，而是基于现有代码能力给出的下一阶段演进建议。

目标是解决下面三个问题：

- 当前“单地址定向分享”体验偏重，批量分享成本高
- 目录共享虽然已有基础能力，但还没有成为主路径
- 一部分资产需要直接面向全体已登录用户共享

## 文档边界

本文只讨论“站内共享能力应该怎样演进”，重点是共享对象、受众模型、分组语义与全员共享。

以下内容不在本文重复展开：

- 个人资产 / 应用资产目录如何分层：见 [资产空间分层设计.md](./资产空间分层设计.md)
- 当前分享与回收站的实现现状：见 [分享与回收站设计.md](./分享与回收站设计.md)
- 具体接口字段示例：见 [WebDAV文件CRUD API（简明版）.md](./WebDAV文件CRUD API（简明版）.md)

## 1. 当前现状

当前代码里实际上已经有两套分享模型：

- 外链分享：`share_items`
  - 面向公开链接访问
  - 当前只支持文件，不支持目录
- 定向分享：`share_user_items`
  - 面向站内已登录用户
  - 已支持文件或目录
  - 已支持 `read/create/update/delete` 权限

对应代码入口：

- 外链分享服务：[internal/application/service/share_service.go](../../internal/application/service/share_service.go)
- 定向分享服务：[internal/application/service/share_user_service.go](../../internal/application/service/share_user_service.go)
- 分享文档：[分享与回收站设计.md](./分享与回收站设计.md)

当前前后端已经落地统一“站内共享创建”语义：

- `targetMode=addresses`：使用 `targetAddresses`，可传多个地址
- `targetMode=groups`：使用 `groupIds`，可传多个分组
- `targetMode=all_users`：共享给所有已登录用户
- 统一调用 `share/user/create`，由后端完成受众展开与去重

对应实现见：

- [web/src/views/home/Index.vue](../../web/src/views/home/Index.vue)

这说明产品方向已经从“多条单用户分享记录”演进到“单条共享对象 + 多受众”模型。

## 2. 当前问题

### 2.1 单地址共享是能力，不应该是主模型

现在的定向分享是“一个路径 -> 一个目标地址 -> 一条分享记录”。

这个模型可以工作，但不适合作为后续主模型，因为：

- 共享给 10 个人，就会创建 10 条几乎重复的记录
- 后续如果要统一修改权限、有效期、备注，需要逐条处理
- 前端列表会出现大量重复路径记录
- “分组共享”只能退化成“前端展开后批量创建 N 条单用户分享”

### 2.2 地址簿分组和共享受众还不是同一个概念

当前地址簿分组的本质是联系人管理工具，不是正式 ACL 主体。

如果直接把地址簿分组等同于长期权限组，会有几个问题：

- 用户只是改了联系人分组，可能意外改变共享权限
- 地址簿是“个人管理视角”，共享受众是“权限控制视角”
- 后续很难解释“改分组到底是改联系人，还是改访问权限”

因此更稳妥的做法是：

- 短期把地址簿分组作为“批量选人器”
- 中长期再决定是否需要独立的“共享组”概念

### 2.3 “面向所有用户共享”不等于“公开链接”

“所有用户共享”和“公开分享链接”是两种完全不同的能力：

- 公开链接：知道链接即可访问，偏向站外分发
- 所有用户共享：必须登录系统，偏向站内公共资源

如果把“所有用户共享”做成公开链接，会带来这些问题：

- 权限边界过粗
- 审计困难
- 撤销和管理不清晰
- 产品心智混乱

所以“所有用户共享”应属于站内共享的一种 audience，而不是公开链接的变体。

### 2.4 目录共享已经有基础，但产品主路径还不够明确

当前定向分享已经支持目录，见：

- [internal/application/service/share_user_service.go](../../internal/application/service/share_user_service.go)

但从产品形态看，目录共享还没有成为默认主路径，用户仍然更容易把分享理解成“给一个地址发一个对象”。

后续如果要提升体验，应该反过来：

- 把目录共享作为主路径
- 单文件共享作为轻量特例

### 2.5 当前分享是路径型绑定，目录变更会影响体验

当前前端删除路径前会检测受影响分享并先撤销：

- [web/src/views/home/Index.vue](../../web/src/views/home/Index.vue)

这说明当前分享本质上还是“路径型引用”，不是独立对象引用。

这个设计在当前阶段是可以接受的，但在做目录共享和公共资源时，需要在产品说明里明确边界：

- 被分享根路径被删除或撤销后，共享随之失效
- 后续若要支持更强的“共享目录重命名/迁移后仍保持稳定”，需要额外设计对象标识或路径迁移机制

当前阶段不建议先碰这个复杂度。

## 3. 推荐路线

推荐把分享能力明确拆成两条产品线：

### 3.1 外链分享

定位：

- 面向匿名或站外访问
- 保留 token / URL 形式
- 继续用于“发一个链接出去”

建议：

- 保留当前 `share_items`
- 短期内继续保持“只支持文件”
- 不把“全员共享”塞进这条模型里

### 3.2 站内共享

定位：

- 面向已登录用户
- 支持文件与目录
- 支持权限控制
- 支持单用户、分组、全员三种受众

建议：

- 让“单地址共享”成为“站内共享”的一个特例
- 让“分组共享”成为“站内共享”的主增强点
- 让“所有用户共享”成为“站内共享”的内建 audience

一句话概括：

- 外链分享解决“发链接”
- 站内共享解决“协作和分发”

## 4. 目标数据模型

推荐不要继续在 `share_user_items` 上堆更多分支语义，而是引入“共享对象”和“共享受众”两层结构。

### 4.1 推荐模型

#### `internal_share_items`

表示“一次站内共享本身”。

建议字段：

- `id`
- `owner_user_id`
- `owner_username`
- `path`
- `name`
- `is_dir`
- `permissions`
- `expires_at`
- `status`
- `created_at`
- `updated_at`

说明：

- `path` 仍然沿用当前路径型模型，先不引入复杂对象引用
- `permissions` 先继续复用当前权限编码方式
- `status` 可先只保留 `active` / `revoked`

#### `internal_share_audiences`

表示“一次站内共享到底发给谁”。

建议字段：

- `id`
- `share_id`
- `audience_type`
- `audience_ref`
- `snapshot_name`
- `permissions_override`
- `created_at`

其中：

- `audience_type` 取值建议为：
  - `user`
  - `group`
  - `all_users`
- `audience_ref`
  - `user` 时存用户 ID
  - `group` 时存分组 ID
  - `all_users` 时为空

### 4.2 为什么推荐拆成两层

因为“共享对象”和“共享对象的受众”不是一个维度：

- 共享对象关心的是：
  - 共享什么
  - 谁创建的
  - 权限和有效期是什么
- 受众关心的是：
  - 这份共享到底给谁看
  - 是用户、分组还是全员

拆开后才能自然支持：

- 一次共享对应多个受众
- 一处改权限，多个受众一起生效
- 一次撤销整个共享，而不是删一堆重复记录
- 后续加审计、备注、标签、排序更容易

### 4.3 关于分组的推荐语义

这里建议分两阶段看：

#### 阶段一：分组只做批量选人器

即：

- 创建共享时可选择地址簿分组
- 系统读取分组当前成员
- 展开成多个 `user` audience

优点：

- 落地快
- 语义稳定
- 不会出现“后来改了联系人分组，历史共享权限也跟着变”的副作用

缺点：

- 后续如果联系人加入/移出分组，不会自动同步历史共享

#### 阶段二：如果业务确实需要，再引入真正的共享组

只有在下面情况成立时，才建议继续往前做：

- 用户明确需要“修改组成员就同步影响共享权限”
- 系统已经具备足够清晰的共享组管理界面
- 审计和风险可接受

在此之前，不建议直接把地址簿分组当成长期 ACL 组。

### 4.4 关于“所有用户共享”

建议作为 `audience_type = all_users` 的内建受众。

这样设计的好处是：

- 权限边界清晰，必须登录才能访问
- 可以和单用户/分组共享共用同一套站内共享机制
- 后续可以单独提供“公共资源”入口，而不是复用外链

## 5. API 草案

### 5.1 设计原则

- 浏览、下载、上传、重命名、删除等运行时接口，尽量复用现有 `share/user` 访问模型
- 变化重点放在“创建 / 列表 / 受众管理”层
- 迁移兼容以“数据回填 + 路径不变”为主，不再保留旧 `targetAddress` 单字段调用兼容

### 5.2 推荐新接口

#### 创建站内共享

`POST /api/v1/public/share/internal/create`

请求示例：

```json
{
  "path": "/docs/project-a",
  "permissions": ["read", "update"],
  "expiresValue": 2,
  "expiresUnit": "week",
  "audiences": [
    { "type": "user", "userId": "u-1" },
    { "type": "user", "walletAddress": "0xabc..." },
    { "type": "groups", "groupIds": ["g-1", "g-2"] },
    { "type": "all_users" }
  ]
}
```

返回示例：

```json
{
  "id": "share-internal-id",
  "name": "project-a",
  "path": "/docs/project-a",
  "isDir": true,
  "permissions": ["read", "update"],
  "expiresAt": "2026-03-31T12:00:00Z",
  "audienceSummary": {
    "userCount": 3,
    "groupCount": 1,
    "allUsers": false
  }
}
```

#### 我发起的站内共享列表

`GET /api/v1/public/share/internal/list?view=owned`

用途：

- 展示我发起的共享对象
- 一条共享对象只显示一行
- 详情中再展开受众信息

#### 分享给我的站内共享列表

`GET /api/v1/public/share/internal/list?view=received`

用途：

- 返回当前用户通过 `user` / `groups` / `all_users` 获得访问权的共享对象

#### 公共资源列表

`GET /api/v1/public/share/internal/list?view=public`

用途：

- 专门列出 `all_users` 类型的共享对象
- 后续可以加标签、搜索、排序、置顶

#### 查看共享受众详情

`GET /api/v1/public/share/internal/audiences?shareId=...`

用途：

- 给拥有者查看这份共享发给了谁
- 支持后续按受众单独撤销

#### 更新共享受众

`PUT /api/v1/public/share/internal/audiences`

请求示例：

```json
{
  "shareId": "share-internal-id",
  "add": [
    { "type": "user", "walletAddress": "0xdef..." },
    { "type": "groups", "groupIds": ["g-2"] }
  ],
  "remove": [
    { "type": "user", "userId": "u-1" }
  ]
}
```

#### 撤销整条站内共享

`POST /api/v1/public/share/internal/revoke`

请求示例：

```json
{ "shareId": "share-internal-id" }
```

### 5.3 兼容策略

现有接口：

- `POST /api/v1/public/share/user/create`
- `GET /api/v1/public/share/user/list`
- `GET /api/v1/public/share/user/received`
- `POST /api/v1/public/share/user/revoke`

当前兼容策略（已落地）：

- 路径保持 `share/user/*` 不变，避免部署和路由层改动
- 创建请求体统一使用新契约：`targetMode + targetAddresses/groupIds`
- 不再兼容旧 `targetAddress` 单字段调用
- 启动迁移会幂等回填 `share_user_items -> internal_share_items/internal_share_audiences`

## 6. 前端交互建议

### 6.1 顶层入口

建议把当前“下载链接 / 定向分享”两个入口调整为：

- `外链分享`
- `站内共享`

这样命名更符合用户认知。

### 6.2 创建站内共享弹窗

建议统一成一套“共享给谁 + 对方能做什么”的表单。

表单建议：

- 共享对象
  - 文件
  - 目录
- 共享范围
  - 指定用户
  - 地址簿分组
  - 所有用户
- 权限预设
  - 只读
  - 可上传
  - 可编辑
  - 完全协作
- 有效期
  - 沿用当前分钟/小时/天/周/月/年

说明：

- “指定用户”和“地址簿分组”应支持多选
- “所有用户”建议单独作为显式选项，不和地址簿混在一起

### 6.3 列表信息架构

站内共享建议拆成三个子视图：

- `我发起的`
- `分享给我的`
- `公共资源`

这样可以避免把“别人分享给我”和“全员共享”混在一个列表里。

### 6.4 详情页建议

对于我发起的站内共享，详情页建议显示：

- 共享根路径
- 是否目录
- 权限
- 过期时间
- 受众摘要
- 受众明细

对于“公共资源”，建议额外支持：

- 简介
- 标签
- 置顶
- 最近更新时间

否则“所有用户共享”很快会退化成难以查找的大列表。

## 7. 分阶段落地顺序

### 7.1 阶段一：先把体验做顺

目标：

- 不大改底层模型，优先提升创建体验

建议动作：

- 保留现有 `share_user_items`
- 前端继续支持“单用户 / 分组”创建
- 把当前分组批量创建能力从“隐藏能力”提升为正式交互
- 列表里对相同路径、多目标的记录做更清晰的聚合展示

说明：

- 当前前端实际上已经具备“分组展开为多个用户”的能力
- 所以阶段一主要是产品和交互收口，不一定要先动数据库

### 7.2 阶段二：落地统一站内共享模型

目标：

- 让“一个共享对象 + 多个受众”成为正式模型

建议动作：

- 引入 `internal_share_items`
- 引入 `internal_share_audiences`
- 让单用户共享、分组共享、全员共享都走同一条创建链路
- 前端列表切到“我发起的 / 分享给我的 / 公共资源”

这是推荐的主实现阶段。

### 7.3 阶段三：再考虑真正的动态组与公共空间运营能力

目标：

- 补更强的组织化共享能力

可选动作：

- 独立“共享组”模型
- 组成员变更影响历史共享的同步策略
- 公共资源标签、搜索、置顶、排序
- 审计日志和访问统计增强

这一阶段不是当前最优先。

## 8. 明确的推荐与不推荐

### 8.1 推荐

- 保留“外链分享”和“站内共享”两条产品线
- 让目录共享成为站内共享主路径
- 让单地址共享退化成站内共享的一个特例
- 短期把地址簿分组作为批量选人器
- 把“所有用户共享”建模成 `all_users` audience

### 8.2 不推荐

- 不推荐把“所有用户共享”直接做成公开链接
- 不推荐继续只围绕 `share_user_items` 叠更多例外逻辑
- 不推荐立刻把地址簿分组等同于长期 ACL 组
- 不推荐一开始就解决“路径重命名后共享仍然稳定”这种高复杂度问题

## 9. 结合当前代码的具体判断

如果只选一条下一步路线，建议优先做：

1. 在产品上把“目录站内共享”提升为主入口
2. 在交互上统一“单用户 / 分组 / 全员”这三种受众
3. 在模型上逐步从 `share_user_items` 迁移到“共享对象 + 共享受众”

这样做的原因是：

- 它和当前已有能力最接近
- 它能覆盖你提出的三个核心诉求
- 它不会把“外链分享”和“站内共享”继续混在一起
- 它给后续公共资源空间、团队分发和权限治理留下了清晰演进路径
